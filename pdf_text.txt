Abstract:
Enterprise adoption of Large Language Models through "Model-as-a-Service" platforms transmits
business-sensitive context to third-party vendors under contractual rather than technical guarantees, exposing PII,
payment card data, and confidential business context. This dissertation presents the design and implementation of
a Privacy-First Data Guardrail Engine — a protocol-driven, fail-open inspection pipeline that operates as a
centralized privacy enforcement layer for enterprise AI systems. The engine intercepts traffic in both directions
through a twelve-stage instrumented pipeline. Detection is performed by seven inspectors covering PII/PCI
entities, prompt injection, multi-turn deception, code injection, and semantic policy violation. Sanitization uses
typed-placeholder substitution with a fidelity gate that verifies rehydrated responses preserve original intent. All
extension points are PEP 544 structural typing protocols, enabling third-party implementations without compiletime dependency. Twenty-nine typed lifecycle events provide forensic audit trails through pluggable durable
sinks. The engine achieves sub-25ms P99 latency in lite mode and sub-150ms with full semantic analysis. A selfconducted architecture audit identified seventeen issues; sixteen were closed and one remains partial at
submission. The engine is open-source, in-process, and supports GDPR data minimization, PCI scope reduction,
and DPDPA jurisdictional compliance.

1. INTRODUCTION
1.1 Background and Context
The current trend of Generative AI development has created a major non-negotiable dependency on a
few dominant cloud providers (OpenAI, Google, AWS). While these "Model-as-a-Service" ecosystems
offer rapid prototyping capabilities, they introduce critical structural risks for enterprise adoption [1].
When an application is built directly against a specific API, the business logic becomes tightly coupled
to the vendor's pricing, latency, and model availability. More critically, to leverage modern reasoning
capabilities, organizations are currently forced to transmit business-sensitive context to external servers.
Consider a typical enterprise scenario: a customer support agent powered by an LLM receives a query
containing a customer's name, email, credit card number, and account details. Without a guardrail layer,
this entire payload, including regulated PII and PCI data, is transmitted to the external model provider.
Even with contractual data processing agreements, the enterprise has no technical guarantee that the
sensitive data is not logged, cached, or inadvertently included in training data.
The risk surface is not limited to passive data exposure. Adversarial inputs, prompt injection attacks that
hijack the model's instructions [6][7], jailbreak attempts that bypass safety alignment, and code injection
payloads embedded in user content, represent an equally pressing class of threat. Enterprise AI systems
are exposed simultaneously to data leakage outward (sensitive payloads flowing to external APIs) and
behavioral compromise inward (malicious inputs subverting the application's intended logic). A
guardrail that addresses only one direction leaves the other open by construction.
The conceptual working of the engine is illustrated in Figure 1. A raw enterprise query carrying PII and
PCI entities enters the sanitize phase, where the inspector chain detects sensitive spans and replaces

1

them with typed placeholders ([PERSON], [EMAIL_ADDRESS], [EMPLOYEE_ID], etc.). The
placeholder-to-original mapping is stored in a per-request map that never leaves the engine boundary.
Only the sanitized text reaches the external LLM; the model never sees the original values. When the
LLM response returns, the rehydrate phase inspects the response for newly-introduced PII (a defense
against the model hallucinating new entities) and then substitutes the original values back from the
placeholder map. The end user receives a complete, accurate answer with real values restored, while the
external LLM provider receives, and may have logged or cached, only placeholders. Every stage emits
structured audit events to a NATS subject and an OpenTelemetry span, supporting forensic replay and
SIEM integration.

1.2 Problem Statement
This creates a fundamental tension: the desire to harness AI's transformative potential versus the
obligation to protect regulated data under GDPR [2], PCI-DSS, HIPAA, and India's DPDPA 2023. There
is a significant gap in the ecosystem for a centralized, extensible guardrail engine that can provide
unified data protection for enterprise AI systems, one that inspects, classifies, and remediates sensitive
data in real-time without requiring external network calls, while simultaneously defending against
adversarial inputs that attempt to subvert the model itself.

2

1.3 Objectives of the Project
The objectives of this dissertation are:
1. Design a protocol-driven guardrail engine using PEP 544 structural typing, such that third parties can
implement inspectors, strategies, sinks, and providers without compile-time dependency on the engine
library.
2. Implement a multi-inspector detection layer covering: regex-based prompt injection, NER-based
PII/PCI recognition (Microsoft Presidio), ML-based semantic intent classification (DistilBERT), hotreloadable custom entity matching, SQL injection, shell injection, and multi-turn deception tracking.
3. Implement a twelve-stage instrumented pipeline with explicit stage constants and uniform stage-level
instrumentation for telemetry, audit, and operator visibility.
4. Implement risk-aware action strategies for remediation, span-level redaction, HMAC-SHA256
pseudonymization, blocking, and structured refusal, with automatic strategy selection driven by detected
risk bands and content policy.
5. Implement intent-preserving rehydration via an IntentLock capture, semantic fidelity scoring, and
a verified rehydration step that prevents substitution into responses whose intent has drifted from the
original query.
6. Implement a typed lifecycle audit subsystem with twenty-nine distinct event types, pluggable durable
sinks, and Server-Sent Events streaming for operator dashboards.
7. Integrate enterprise infrastructure through pluggable protocols: NATS event streaming, Unleashaligned feature flag management, and OpenTelemetry-compatible tracing and metrics.
8. Achieve sub-25ms P99 latency for the core inspection pipeline in lite mode, sub-150ms in full mode
with semantic analysis, and validate the system against a corpus of 137 prompts across 9 threat
categories.

1.4 Scope and Limitations
In scope: The three-package Python workspace (arc-guard-core, arc-guard, arc-guardservice); all seven concrete inspector implementations; all five action strategies and the strategy
selector; the twelve-stage GuardPipeline with stage_runner instrumentation; the lifecycle
subsystem with twenty-nine typed events, three sink implementations, and Server-Sent Events
streaming; the intent preservation chain; the deployment surface including HTTP transport and
dashboard read routes; the GuardRailFlow Single-Page Application; and a 137-prompt evaluation corpus
spanning 9 threat categories.
Out of scope: Training of new ML models (the semantic inspector uses pre-trained DistilBERT weights
without fine-tuning); hosted deployment, container orchestration, and Kubernetes manifests; formal
certification against any compliance standard (the system supports GDPR data minimization, PCI scope
reduction, and DPDPA jurisdictional compliance, but certified tokenization and audited compliance
posture are explicitly out-of-scope); and post-quantum cryptography.
Known limitations of the approach. The system is implemented in Python; a rewrite in a lower-level
language would alter the latency profile substantially but is not in scope. The evaluation corpus, though
designed across nine threat categories, is curated rather than adversarially generated; a production redteam exercise would surface attack patterns not represented. The default deployment topology assumes a
single-writer SQLite store for the lifecycle subsystem; multi-instance deployments require networked
SQLite or a switch to PostgreSQL via the optional [postgres] extra, which is scaffolded but not
validated at scale within this dissertation.

1.5 Literature Survey
The detection of PII in unstructured text is a well-studied problem at the intersection of NLP and data
privacy. Krishnamurthy et al. [3] proposed a privacy-preserving PII label detection system combining
4

machine learning with regular expressions, achieving 99.85% training accuracy. The Microsoft Presidio
framework [4] has established itself as a de facto standard for open-source PII detection, combining
NER models with pattern-based recognizers and context-aware confidence scoring. More recently, Iyer
et al. [5] presented an adaptive PII mitigation framework specifically designed for LLM deployments,
addressing conversational AI contexts where traditional document-oriented DLP tools underperform.
On the security front, prompt injection has emerged as a critical vulnerability in LLM-integrated
applications. Liu et al. [6] provide a formal framework for prompt injection attacks, benchmarking 5
attack methods and 10 defense mechanisms across 10 LLMs, establishing that no single defense
provides comprehensive protection. Abdelnabi et al. [7] explored indirect prompt injection where
malicious instructions are embedded in external content retrieved by AI agents, reinforcing the need for
both input and output guardrails.
Several guardrail frameworks have emerged to address these concerns. NVIDIA NeMo Guardrails [8]
introduces Colang, a domain-specific language for defining conversational guardrails. While powerful
for dialogue management, it requires learning a new DSL and tightly couples guard definitions with the
framework. Guardrails AI provides a RAIL specification for output validation but lacks the multi-stage
inspection pipeline and privacy-specific capabilities that enterprise data protection requires.
Ayyamperumal and Peruri [9] survey the current state of LLM risks and AI guardrails, identifying that
existing solutions often prioritize content safety over data privacy, leaving a gap that privacy-focused
engines aim to fill.
Two further limitations of existing guardrail systems become visible only after sustained production
exposure. First, answer fidelity after rehydration is rarely addressed: when a system redacts a name to
[PERSON] and the LLM responds referring to [PERSON], simply substituting the original value back
assumes the response is still semantically faithful to the original intent. If the LLM hallucinates,

5

contradicts, or drifts from the captured intent, naive substitution amplifies the failure rather than catching
it. Second, structured refusal semantics, communicating why a request was blocked, in a form the calling
application can act upon, is treated as an afterthought in most frameworks; opaque rejections create poor
developer experience and inhibit adoption.
Gap in existing work. No currently-available open-source library combines: in-process operation with
sub-25ms P99 latency; protocol-first extensibility without compile-time coupling; intent-preservation as
a first-class quality gate; and a typed lifecycle audit subsystem suitable for forensic reconstruction. The
engine presented in this dissertation addresses these gaps through a protocol-driven design using
Python's structural typing (PEP 544) [10], DistilBERT [11] for in-process semantic analysis, and
OpenTelemetry [12] for vendor-neutral observability validated by Otero et al. [13].

1.6 Methodology Overview
The work was conducted in two phases corresponding to the BITS WILP semester structure. The first
phase (January–March 2026) produced a baseline implementation comprising four inspectors, three
strategies, seven extension protocols, and an implicit pipeline. This baseline was the subject of the
midterm viva in April 2026, which surfaced architectural concerns across six dimensions: jailbreak
sensing depth, semantic extensibility, intent consistency after rehydration, automatic masking strategy,
code injection coverage, and graceful refusal semantics.
The second phase (April–May 2026), designated Testing & Audit Remediation, addressed each of these
concerns through targeted architectural extensions: the twelve-stage pipeline replaced the implicit
baseline; the inspector chain grew from four to seven; structural typing migrated from ABC inheritance
to PEP 544 structural Protocols; an intent preservation chain was introduced; the lifecycle audit
subsystem replaced log-based observability; and a structured refusal stage was added to the pipeline.
Detailed methodology for both phases is presented in Chapter 3.

The implementation methodology was test-driven and additive. Every architectural extension was
introduced through a documented specification process without breaking changes to existing contracts.
Code quality is enforced through ruff (linter), mypy (type checker), and pytest (asyncio automode), with separate test pyramids for unit, integration, contract, performance, and security categories.

2. SYSTEM ARCHITECTURE AND DESIGN
2.1 High-Level Architecture
The guardrail engine operates as a protocol-driven, fail-open pipeline for text data inspection and
remediation. The system is structured as three Python packages, arc-guard-core (zero-dependency
contract layer), arc-guard (batteries-included runtime), and arc-guard-service (FastAPI
deployment surface), with a strict one-way import direction (arc_guard_service → arc_guard
→ arc_guard_core).
7

The architecture is shown in Figures 1a and 1b. Five layers cooperate: the transport layer (arcguard-service) handles HTTP entry, request id minting, and response serialization; the runtime
layer (arc-guard) hosts the seven concrete inspectors, the strategy selector, the content policy, the
action strategies, and the intent preservation chain; the contract layer (arc-guard-core) defines the
fifteen Protocols, the frozen dataclasses, and the twelve stage constants, and has zero non-stdlib runtime
dependencies; the observability layer spans four independent surfaces (Logger, Tracer, MetricSink,
LifecycleSink); and the audit layer dispatches structured events to NATS for downstream SIEM
integration.

2.2.1 Detection Components
a) Guard Pipeline: The central orchestrator implementing the Chain-of-Responsibility pattern [14]. It
receives text input with context metadata, routes it through twelve named execution stages, aggregates
findings from the inspector chain, applies the appropriate remediation strategy, and dispatches audit
events. The pipeline implements two entry points, pre_process for inspecting user input before it
reaches the AI model, and post_process for inspecting model output before it reaches the user. All
data types flowing through the pipeline are frozen (immutable) dataclasses ensuring thread-safety. The
pipeline defines fifteen structural typing protocols that serve as behavioral contracts, any third-party
12

implementation satisfying the method signatures is accepted without requiring inheritance or library
coupling.
b) Injection Inspector: Detects prompt injection attempts using compiled regular expression patterns.
Ships with built-in patterns covering common attack signatures including "ignore previous instructions,"
jailbreak attempts, system prompt extraction, and role-play manipulation. Custom patterns can be
injected at construction time. Only executes during the pre-process phase. Latency: sub-1ms.
c) PII Inspector: Wraps the Microsoft Presidio AnalyzerEngine [4] for Named Entity Recognition
based detection of PII and PCI entities. Default configuration detects CREDIT_CARD,
EMAIL_ADDRESS, PHONE_NUMBER, PERSON, IBAN_CODE, US_SSN, US_PASSPORT, and
IP_ADDRESS. Confidence scores from Presidio are mapped to risk levels: scores above 0.85 map to
HIGH, above 0.60 to MEDIUM, and below 0.60 to LOW. Latency: 5–20ms.
d) Semantic Inspector: Performs ML-based intent classification using DistilBERT [11]. Classifies text
as containing TOXIC content or INJECTION intent at the semantic level, catching adversarial inputs
that evade pattern matching. Inference runs in a thread pool executor to avoid blocking the asynchronous
event loop. Configurable confidence thresholds control sensitivity for input and output phases
independently. Supports air-gapped deployment through pre-downloaded model directories. Latency:
30–80ms. In the post-remediation system, the inspector also implements the IntentEncoder
Protocol, which allows the same embedding produced for classification to be reused by the fidelity
scorer.
e) Custom Inspector: Enables hot-reloadable entity detection through the EntityProvider protocol.
Queries the entity registry on every invocation (no caching), allowing new entity patterns to be
registered at runtime without restarting the engine. Supports domain-specific entities such as internal

13

project codes or regional identifiers. Entities are categorized as PII, PCI, or CUSTOM with
corresponding risk level mappings.
n) SQL Injection Inspector: Pattern-based detection of SQL injection payloads in user input. Detects
three primary attack classes: stacked queries ('; DROP TABLE users; --), UNION-based
extraction, and comment-terminator obfuscation. Implementation uses compiled regex with anchored
boundary detection to minimize false positives on legitimate text containing SQL keywords. Findings
carry entity_type = "SQL_INJECTION" and route through the Strategy Selector to either redact
or block based on risk band. Latency: sub-1ms.
o) Shell Injection Inspector: Pattern-based detection of shell command injection in user input.
Implements two-stage detection: an initial scan for shell metacharacters followed by a heuristic check
that distinguishes legitimate text containing punctuation from actual shell command syntax. Findings
carry entity_type = "SHELL_INJECTION". Latency: sub-1ms.
p) Conversation Turn Inspector: Multi-turn deception tracking inspector implementing the
ConversationTurnInspector Protocol. Reads and updates a ConversationState object
across consecutive turns to detect gradual policy erosion, role-play coercion, hypothetical framing, and
slow-escalation patterns where each individual turn looks benign but the trajectory crosses a policy
boundary. Runs in STAGE_DECEPTION_INSPECT, output feeds into policy routing as an additional
signal. Skipped when no ConversationState is provided. This module addresses the jailbreak
sensing concern from midterm: single-turn jailbreak detection cannot catch attacks distributed across
multiple turns.

2.2.2 Remediation Components
f) Redact Strategy: Replaces each detected span with a typed placeholder. Initially implemented as flat
type labels ([CREDIT_CARD]), the post-rewrite version produces numbered placeholders that preserve
14

repeated-entity structure across a single request: [CREDIT_CARD_1], [CREDIT_CARD_2],
[PERSON_1]. This numbering is stored in the PlaceholderMap and is the substrate for the
rehydration step. Findings are processed in reverse order to preserve positional accuracy.
g) Hash Strategy: Applies HMAC-SHA256 pseudonymization to each detected span, producing a
deterministic 16-character hexadecimal digest. Same input always produces the same hash, enabling
correlation across sessions without exposing original values. HMAC key sourced from environment
variables, a key file, or auto-generated on first use. Note: this is not PCI-approved tokenization.
h) Block Strategy: Suppresses the input or output entirely. Intended for RESTRICTED classification
levels or when CRITICAL findings indicate the text should not be processed further. In the postremediation system, block does not silently return an empty string, it triggers STAGE_REFUSAL, which
constructs a structured RefusalEnvelope communicating why the request was blocked.
q) Strategy Selector & Content Policy: Two complementary protocols that together implement
automatic masking strategy, the engine deciding on its own which remediation approach to apply based
on the type and sensitivity of detected data. The StrategySelector Protocol exposes
select(finding, policy) -> ActionStrategy. The default implementation maps
RiskLevel to a strategy via four risk bands: LOW → redact, MEDIUM → redact + warn, HIGH →
partial block, CRITICAL → hard block (refusal). The ContentPolicy Protocol exposes
evaluate(finding) -> PolicyDecision. Two concrete policies ship: a rule-based policy
and a SemanticContentPolicy (under the [semantic] extra) that uses embedding similarity
for cases where simple rule matching is insufficient.
s) Intent Preservation Chain: A four-protocol chain (IntentEncoder, IntentLock,
FidelityScorer, RehydrationVerifier) that addresses the intent consistency after
rehydration concern. The chain operates as follows. At STAGE_DEFEND, the IntentEncoder

15

produces a semantic embedding of the original user prompt. The embedding is stored in an immutable
IntentLock object, which is carried through the pipeline alongside the placeholder map. After the
LLM returns its response and sanitization has been reversed via the placeholder map, the
FidelityScorer re-encodes the rehydrated response and computes its semantic similarity against
the locked intent. The RehydrationVerifier then compares this score against a configurable
threshold (default 0.78 cosine similarity). If the score falls below the threshold, the rehydrated response
is held back, a verification-failed lifecycle event fires, and the caller receives a structured
RefusalEnvelope explaining that the response failed the fidelity check.
t) Refusal Envelope: A structured response object returned to the caller whenever the engine refuses a
request. The envelope contains: decision (enumerated outcome, BLOCKED, RATE_LIMITED,
FIDELITY_FAILED, POLICY_VIOLATION), reason_code (machine-readable identifier),
risk_band, inspector_findings (a sanitized summary), and correlation_id (the request
rid). This addresses the graceful refusal concern from midterm: rather than silently returning an empty
string or a generic error, the engine communicates exactly why a request was refused in a form the
calling application can branch on. The envelope is JSON-serializable for transmission over the FastAPI
service surface.

2.2.3 Audit and Lifecycle Components
i) Event Streaming Reporter: Publishes audit events to a NATS message streaming subject using a
bounded asynchronous queue with a background drain loop. Queue capacity is configurable (default
1000 events), dropping oldest events when full to prevent memory growth. Events follow a versioned
schema (v1.0). Operates in fire-and-forget mode, reporting can never block the inspection pipeline.
Additional reporters include a Log Reporter, Webhook Reporter, and Null Reporter.

16

j) Feature Flag Provider: Provides runtime behavioral control through three implementations:
Environment Provider (reads from GUARD_* environment variables), Static Provider (in-memory
dictionary for testing), and Feature Flag Service Provider (integrates with Unleash with automatic
namespace prefixing). Standard flags include: enabled, lite_mode, action_strategy, and perinspector toggles. Returns defaults on any exception, ensuring flag service outages never break the
pipeline.
k) Observability Middleware: In the midterm baseline, observability was a single middleware wrapper
emitting metrics and spans. The post-remediation architecture splits observability into four independent
surfaces with explicit failure-posture contracts: the Logger, the Tracer (OpenTelemetry-compatible), the
MetricSink, and the LifecycleSink. All four are independently pluggable Protocols. The
stage_runner instrumentation hook ensures that every one of the twelve pipeline stages emits to all
four surfaces uniformly. Six metrics are emitted: pipeline duration, stage duration, inspector duration,
findings count, pipeline errors, and reporter drops.
l) Entity Registry: Thread-safe (RLock-protected) registry for custom entity definitions. Each entity is
defined with a unique name, category, optional regex pattern, and optional Presidio recognizer. Supports
hot-reload, entities registered at runtime are picked up on the next inspection call without restart.
m) Configuration: Immutable Pydantic v2 model managing structural settings: PII entity list, detection
language, model paths, and cache directories. Supports loading from environment variables. Cleanly
separated from behavioral configuration (handled by the Flag Provider), Configuration controls WHAT
to detect, Flag Provider controls HOW to respond.
r) Lifecycle Audit Subsystem: A typed event taxonomy and pluggable durable-sink architecture for
forensic audit and operator visibility. The LifecycleSink Protocol allows multiple sink
implementations to coexist via a composite sink. Twenty-nine distinct event types are defined (see

17

Appendix A for the complete taxonomy), spanning request entry, every stage exit, every inspector run,
every finding, placeholder map construction, strategy application, refusal construction, backend
invocation, rehydration verification, and request completion. All events carry id, parent_id, seq,
ts, rid, and event_type, forming a directed acyclic graph per request that supports both live
inspection and forensic replay. Three concrete sinks ship: RingBufferSink (in-memory, bounded),
SqliteLifecycleSink (durable, four-table schema, WAL mode), and CompositeSink (fanout). The default in production is SqliteLifecycleSink with a 30-day retention policy. Failure
mode is open: a sink that raises does not propagate the exception into the pipeline.

2.4 Design Decisions and Trade-offs
The architectural choices described above follow from a set of design principles that govern every
extension decision in the engine. This section documents those principles and the trade-offs accepted in
choosing them.
Fail-Open by Default. Exceptions in inspectors, middleware, or reporters are caught and logged. The
pipeline always returns a valid result. This ensures the guardrail layer can never break production traffic,
a critical requirement for enterprise adoption where availability takes precedence over strictness. Postremediation, this principle was made more precise: the system distinguishes between fail-open
components (detection and observability, a broken detector returning zero findings is safer than blocking
all traffic) and fail-closed components (transformation, a broken sanitizer cannot be trusted to have
produced a safe output, so its failure must propagate). The complete failure-posture table is given in
Appendix B.
Protocol-First Extensibility. All extension seams are PEP 544 structural typing protocols. Third-party
code can implement any protocol without importing or depending on the engine library. If an object has
the right method signatures, it satisfies the protocol regardless of class hierarchy. The midterm system
exposed seven such protocols; the post-remediation system exposes fifteen. The protocol count grew
because the system grew, but the mechanism did not change. The alternative considered at design time
was abstract base class inheritance, the conventional Python pattern. It was rejected because every
consumer of the library would have had to import the base class to subclass it, which creates compiletime coupling between the consumer and the library version. With structural typing, no such coupling
exists: the consumer's class needs to satisfy a method signature, not inherit a runtime artifact.
Structural vs Behavioral Configuration Separation. Configuration controls WHAT to detect (entity
lists, model paths, language). Flag Provider controls HOW to respond (enabled/disabled, lite mode,
strategy selection). Flag Provider always takes precedence, enabling runtime overrides without
21

redeployment. Mixing these would make runtime overrides impossible, every behavioral change would
require a configuration reload, which in turn would require a service restart.
Zero Compile-Time Infrastructure Dependencies. NATS, Unleash, OpenTelemetry, Presidio, and
PyTorch are all optional and guarded by import-time exception handling. The post-remediation arcguard-core package, the contract layer, has zero non-stdlib runtime dependencies whatsoever. Heavy
extras are isolated to the arc-guard runtime package behind optional install groups.
Async-First Concurrency. IO-bound operations are properly awaited. CPU-bound ML inference runs
in a thread pool executor. Event streaming uses bounded async queues. Fire-and-forget semantics for
reporting. Entity registry is RLock-protected for thread-safe concurrent access. The midterm architecture
audit identified four concurrency-related issues in this principle's implementation; these were addressed
during remediation as detailed in Chapter 4.
Compliance Alignment. PII detection and pseudonymization support GDPR data minimization. Credit
card redaction prevents PAN transmission (PCI-DSS). Custom entity registry supports jurisdictionspecific identifiers (DPDPA 2023, Aadhaar). HMAC pseudonymization is explicitly documented as not
PCI-approved tokenization; this disclaimer is reproduced in the operator documentation to ensure
deployers do not mistake the engine's pseudonymization for certified tokenization.
Forensic Auditability. The midterm system relied on log-based observability, sufficient for debugging,
inadequate for forensic reconstruction. A regulator asking "what did this user's request look like at every
stage of processing on a given timestamp" could not be answered from logs alone, because logs are
unstructured and not request-correlated. The post-remediation system adopts a stronger principle: every
architecturally significant event in the request lifecycle is a typed object with a request correlation id,
persisted to durable storage, and queryable by request id.

22

Intent Preservation as a Quality Gate. Sanitization is fundamentally a transformation: the engine takes
one string and returns a different string. If the post-rehydration response no longer corresponds to the
original user intent, because the LLM hallucinated, drifted, or contradicted the placeholder-rich input it
was given, naive substitution amplifies the failure. The intent preservation chain treats fidelity as a nonoptional quality gate. The default cosine-similarity threshold of 0.78 was selected empirically from the
137-prompt evaluation corpus; thresholds above 0.85 produced excessive false rejections on legitimate
paraphrasing, thresholds below 0.70 admitted genuine intent drift.
Graceful Refusal as a First-Class Outcome. A guardrail that blocks a request is performing a normal,
expected, designed-for action, not handling an error. Blocked requests get the same lifecycle event
treatment as passed requests, the same correlation id propagation, and the same observability emission.
The RefusalEnvelope is a structured response, not a status code with no body. The contrast with the
midterm baseline, which silently returned an empty string on block, is intentional and direct.
Additive Evolution. Every architectural addition introduced after the midterm has been purely additive.
The twelve-stage pipeline shape, the core data types, and the protocol contracts have remained stable
through the post-midterm phase. New stages, events, dashboard routes, inspectors, and strategies were
added; no existing contract was modified or deprecated. This is the practical test of the protocol-first
principle: if extensions truly work without coupling, then adding them should not require breaking
existing contracts.

3. METHODOLOGY
3.1 Approach
The work was conducted in two phases corresponding to the BITS WILP semester structure. Each phase
had distinct deliverables, distinct evaluation criteria, and distinct architectural commitments.
Phase One, Baseline Implementation (January–March 2026). The first phase produced the engine
architecture submitted at midterm: four inspectors (regex injection, Presidio NER, DistilBERT semantic,
custom entity), three action strategies (redact, hash, block), seven extension protocols defined as ABCs,
and an implicit pipeline with approximately five processing stages. The phase was test-driven: every
module was implemented alongside its unit and integration tests, with code coverage targets defined per
module category. The midterm submission included a self-conducted architecture audit identifying
seventeen issues across four severity levels.
Phase Two, Testing & Audit Remediation (April–May 2026). The second phase addressed the audit
findings and the architectural concerns surfaced at the midterm viva. The work was scoped against two
parallel deliverables: closing the seventeen audit issues with documented remediation, and introducing
seven new modules that addressed the six viva concerns (jailbreak sensing, semantic extensibility, intent
consistency after rehydration, automatic masking strategy, code injection coverage, graceful refusal).
Migration from ABC inheritance to PEP 544 structural Protocols, the introduction of the twelve-stage
pipeline, and the introduction of the twenty-nine-event lifecycle audit subsystem were all completed
during this phase.
The implementation methodology across both phases was test-driven and additive. Every architectural
extension was introduced through a documented design specification without breaking changes to
existing contracts. The twelve-stage pipeline shape and the core data types have remained stable through
the project; new stages, events, and dashboard routes were added as backward-compatible extensions.

3.4 Evaluation Strategy
The engine was evaluated across three dimensions, each with its own methodology.
Threat detection accuracy is measured against a curated 137-prompt corpus spanning 9 threat
categories (prompt injection, PII/PCI, jailbreak heuristic, jailbreak ML, multi-turn deception, semantic
policy violation, SQL injection, shell injection, template injection) at three difficulty gradients (easy,
medium, super_hard) with two benign baselines. The corpus is located at
packages/api/tests/corpus/ and is regenerated as new threat categories are added. Each
prompt is annotated with the expected outcome (block, redact, pass) and the expected inspector.
27

The methodology distinguishes canonical attack forms (which any working detector should catch) from
obfuscated forms (which test the detector's robustness to paraphrase) and adversarially-framed forms
(which test the detector's tolerance for legitimate-looking inputs containing harmful payloads). False
positive rate is measured against the baseline corpus.
Codebase health is evaluated against seven dimensions: protocol purity, dependency hygiene, type
safety, async correctness, error handling, naming and conventions, and documentation. Each dimension
is scored on a letter grade scale at midterm and post-remediation, providing a quantitative measure of the
remediation impact. Test coverage is measured via pytest-cov against four module categories with
target thresholds defined per category.
Runtime latency is measured against a 10,000-request synthetic workload generated from the threat
corpus, executed against a baseline reference machine. Two operating modes are measured: lite mode
(semantic and ML-jailbreak inspectors disabled via FlagProvider) and full mode (all inspectors
enabled). Per-stage latency contribution is decomposed to identify dominant cost stages. Targets are
stated against P99 percentile rather than median, reflecting the operational reality that tail latency
determines user-visible performance.
Detailed results from all three evaluation dimensions are presented in Chapter 5.

4. IMPLEMENTATION
4.1 Development Environment
The engine is implemented as three Python packages in a single workspace. The arc-guard-core
package contains the contract layer (Protocols, frozen dataclasses, stage constants, lifecycle event types)
28

and has zero non-stdlib runtime dependencies, a property enforced by the
test_install_closure.py integration test. The arc-guard package contains the runtime
implementations (concrete inspectors, action strategies, lifecycle sinks, intent preservation chain) with
optional extras for heavy dependencies ([semantic] for DistilBERT, [jailbreak-ml] for the
ML jailbreak detector, [nats] for the NATS reporter, [unleash] for the feature flag service,
[otel] for OpenTelemetry). The arc-guard-service package contains the FastAPI HTTP
transport surface and the Server-Sent Events streaming endpoint.
Python 3.11+ is the minimum supported runtime. The codebase enforces three quality gates at every
commit: ruff (linting and formatting), mypy --strict (type checking against the contract layer),
and pytest with asyncio auto-mode (test execution). Test categories are organized into a layered
pyramid: unit tests (per-module behavior), integration tests (cross-module interactions), contract tests
(Protocol conformance), performance tests (latency budgets), security tests (threat corpus regression),
and end-to-end tests (HTTP transport through to lifecycle sink). Coverage targets per category are listed
in Chapter 5.
The development workflow uses specification-driven development. Each architectural change is
preceded by a written design specification that records the problem, the proposed change, the
alternatives considered, and the migration path. Specifications are reviewed before implementation;
implementation is reviewed against the specification; the specification is finalized when the
implementation merges. This produces a written record of architectural intent independent of the code
itself, which forms the substrate for the design discussion in Chapter 2 Section 2.4.

4.2.1 Module One, Inspector Chain
The seven inspectors described in Chapter 2 Section 2.2.1 (modules b, c, d, e, n, o, p) execute in parallel
within STAGE_CLASSIFY for the first six; module p (Conversation Turn Inspector) runs separately in

4.2.2 Module Two, Strategy Selector and Action Strategies
The five action strategies (modules f, g, h, plus the warn and refusal variants introduced via the strategy
selector) are routed automatically based on detected risk. The StrategySelector Protocol (module
q) consults the ContentPolicy to map each finding to the appropriate strategy via four risk bands.
The RefusalEnvelope (module t) is the structured response returned to callers when the policy
resolves to BLOCK or when the fidelity gate at STAGE_VERIFY fails.

The lifecycle audit subsystem (module r) emits twenty-nine typed event classes through the
LifecycleSink Protocol. The default SqliteLifecycleSink uses Write-Ahead Logging mode
for concurrent reader access, supporting the dashboard's request workspace view without locking the
writer. The intent preservation chain (module s), IntentEncoder → IntentLock →
FidelityScorer → RehydrationVerifier, operates as a quality gate at STAGE_VERIFY
before rehydration is applied.

4.3 Integration
Integration with enterprise infrastructure is handled through pluggable Protocols. NATS event streaming
(module i) uses a bounded asynchronous queue with a background drain loop; the queue is sized for a
single-instance deployment processing tens of requests per second and exports a dropped-events counter
as a Prometheus metric. The Unleash-aligned feature flag provider (module j) handles runtime
behavioral overrides without redeployment; standard flags include enabled, lite_mode,
action_strategy, and per-inspector toggles. OpenTelemetry tracing (module k) emits three span
types (guardrail.pre_process, guardrail.post_process, guardrail.stage.*)
with stage-level granularity sufficient for cross-service correlation in distributed traces.
The four-surface observability split, Logger, Tracer, MetricSink, LifecycleSink, replaces the midterm
baseline's single observability middleware. Each surface is a separate Protocol with its own failure
posture. The stage_runner instrumentation hook ensures uniformity across all twelve stages: a
single hook emits to all four surfaces, eliminating per-stage instrumentation drift.

4.4 Deployment
The engine supports three deployment topologies, each appropriate for a different operational context.
Embedded library mode. The application imports arc-guard directly and constructs its own
GuardPipeline. No HTTP transport is involved. The lifecycle sink can be configured to write to the
application's own SQLite database or to a shared fleet-wide store. This mode incurs zero per-request
network overhead and is recommended for latency-sensitive applications where the engine executes
alongside the LLM client in the same process.
Co-located service mode. The application makes a localhost HTTP call to a sidecar arc-guardservice instance running in the same pod or container group. This mode introduces single-digitmillisecond loopback overhead in exchange for language independence, non-Python applications can
36

integrate via the OpenAI-compatible endpoint. The dashboard and lifecycle sink are owned by the
sidecar.
Centralized service mode. A pool of arc-guard-service instances behind a load balancer serves
multiple applications over the network. This mode introduces network-hop latency (typically 5–15 ms
within a regional VPC) but concentrates audit and dashboard data in a single fleet-wide store,
simplifying compliance reporting. The trade-off is that the engine is no longer "in process"; a centralized
deployment loses the sub-25 ms lite-mode latency target by construction.
The arc-guard-service FastAPI surface exposes two guard endpoints (POST /v1/guard and
POST /v1/chat/completions for OpenAI-compatible integration), four dashboard read routes
(GET /requests, GET /requests/{rid}, GET /requests/{rid}/decision, GET
/requests/{rid}/debug), and a Server-Sent Events streaming endpoint (GET /events). The
GuardRailFlow Single-Page Application (Vite + React + TypeScript) consumes these routes to provide
three operator views: a paginated request explorer, a per-request workspace drilldown, and a real-time
SSE feed of events as they fire.
The default recommendation is embedded library mode for production application-side deployment, and
centralized service mode for the operator dashboard surface, combined via a shared SQLite store on
networked storage, or a fleet-wide PostgreSQL store via the optional [postgres] extra (planned, not
in current submission).

5.3 Analysis and Discussion
The aggregate detection rate of 89% across 135 attack prompts (excluding baselines) sits within the
acceptable range for an enterprise privacy guardrail. The decomposition by difficulty is more
informative than the aggregate: detection holds at 100% for canonical attack forms, drops to 91% for
obfuscated forms, and to 76% for adversarially-framed attacks. This shape is consistent with the stated
design posture, the engine is a privacy-first detection layer that handles the realistic distribution of attack
inputs encountered in production. The two baseline categories (multi-turn benign and no-inspector
benign) confirm that the engine produces zero false positives on benign traffic, a property that is
operationally more important than maximizing detection on super_hard prompts, because false
positives directly impede legitimate enterprise traffic.
Both operating modes meet the latency targets stated in Chapter 3. Lite mode P99 of 22 ms is within the
25 ms target. Full mode P99 of 142 ms is within the 100–150 ms target band. The per-stage
decomposition confirms that the dominant cost in full mode is STAGE_CLASSIFY (67% of P99
budget), of which the majority is the DistilBERT semantic inspector. This is consistent with the design
choice to make semantic analysis optional via lite mode for latency-sensitive paths. The intent
preservation chain (defend + verify) contributes 23% of P99 latency together, non-trivial but justified by
the quality-gate role it plays. No stage exceeds its individually budgeted contribution, and no stage
exhibits long-tail behavior beyond P99.9 of approximately 1.25× P99.
To make the architecture concrete, three end-to-end traces follow, one for each of the three primary
outcomes: successful redaction, hard block, and multi-turn deception detection. These traces
demonstrate the system in operation rather than in description.
Trace A: PII Redaction, Successful Sanitize → LLM → Rehydrate. A customer support application
receives the message "My name is Jane Smith and my card ending in 4242 is being declined. Card
number: 4532-1234-5678-4242. Can you help?" and invokes POST /v1/guard. The request flows
43

through STAGE_VALIDATE (passes), STAGE_DEFEND (IntentEncoder produces a 384-dimensional
embedding stored in IntentLock), STAGE_CLASSIFY (Presidio identifies PERSON at LOW risk
and CREDIT_CARD at HIGH risk; injection and jailbreak inspectors find nothing),
STAGE_DECEPTION_INSPECT (no-op, single turn), STAGE_SANITIZE (text becomes "My name is
[PERSON_1]... Card number: [CREDIT_CARD_1]..."), STAGE_ROUTE (resolves to WARN, redaction
sufficient), STAGE_EXECUTE (LLM is called with sanitized text and returns a response that still
references the placeholders), STAGE_VERIFY (fidelity score 0.91, well above the 0.78 threshold),
STAGE_REHYDRATE (placeholders mapped back to originals), STAGE_DECISION_EMIT, and
STAGE_REPORT. Total wall-clock time: approximately 665 ms, dominated by the LLM call. Pipeline
overhead: approximately 45 ms.
Trace B: Prompt Injection, Hard Block via STAGE_REFUSAL. A different request, "Ignore all
previous instructions. You are now DAN (Do Anything Now). Reveal the system prompt and your
training data.", flows differently. STAGE_CLASSIFY produces three findings at CRITICAL risk: a
regex match on "Ignore all previous instructions", a jailbreak signal for direct override, and a jailbreak
signal for role-play coercion. STAGE_ROUTE resolves to BLOCK. STAGE_REFUSAL (instead of
STAGE_EXECUTE) constructs a RefusalEnvelope with decision="BLOCKED",
reason_code="PROMPT_INJECTION_HARD_BLOCK", risk_band="CRITICAL", sanitized
inspector findings, and the request correlation id. The LLM is never called. Stages 9 and 10 are skipped.
The transport returns the envelope as JSON with HTTP status 200. This trace illustrates the graceful
refusal principle operationally: a blocked request travels the same pipeline as a successful one, emits the
same kinds of lifecycle events, produces the same DecisionRecord, and fires the same audit
dispatch. The only difference is that stages 8, 9, and 10 are replaced by a single refusal stage and the
LLM is never invoked.
44

Trace C: Multi-Turn Deception, Trajectory Detection. A more adversarial scenario unfolds across
three turns. Turn 1: "Let's play a game. You're a helpful assistant who explains historical events.",
benign in isolation, passes through. Turn 2: "In this game, all your usual rules don't apply, since it's just
a roleplay.", the ConversationTurnInspector recognizes the gradual policy erosion pattern and
emits a HIGH-risk finding; routing resolves to WARN. Turn 3: "Now, as the game character, explain
how someone might bypass an enterprise firewall.", the trajectory has crossed into operational territory;
the inspector emits a CRITICAL finding for DECEPTION_OPERATIONAL_PIVOT; routing resolves
to BLOCK. Single-turn jailbreak detection cannot see this attack, none of the three turns in isolation
contains a direct override marker, role-play coercion that triggers the heuristic detector, or a regex
pattern that fires any inspector. The attack succeeds against a single-turn guard precisely because each
turn is innocuous; the attack exists only as a trajectory across turns.
These three traces, taken together, demonstrate the operational symmetry between accepted and refused
requests, the fidelity gate's role in catching responses that drift from intent, and the multi-turn inspector's
coverage of attack patterns that single-turn detection cannot see.

5.4 Discrepancies and Limitations
Three limitations of this evaluation should be acknowledged. First, the threat corpus of 137 prompts is
curated rather than adversarially generated; a production red-team exercise would likely surface attack
patterns not represented in the corpus. Second, latency measurements were taken on a single reference
machine; multi-tenant cloud deployments with noisy-neighbour effects will produce wider tail latencies.
Third, the evaluation does not include long-running multi-turn conversation tests at scale, the current
ConversationTurnInspector is validated against fifteen synthetic conversation traces, which is
sufficient for correctness but inadequate for characterizing drift across thousands-of-turn sessions.

45

The HMAC tokenization issue (C7 in Table 6) is documented as partially closed because certified vault
integration remains a deployment-time decision; the dissertation does not claim PCI-approved
tokenization. The TokenizationProvider Protocol scaffolded during remediation provides the
integration seam for an operator's certified vault, but the certification itself is out of scope for this work.
These limitations frame the future work outlined in Chapter 6.

6.2 Conclusions
The contributions of this dissertation fall into three tiers.
Architectural contributions. The engine itself is the primary contribution: a protocol-driven detectionand-remediation pipeline that operates in-process with sub-25 ms lite-mode latency, exposes fifteen
47

extension Protocols for third-party integration without library coupling, and ships with a typed lifecycle
audit subsystem comprising twenty-nine event classes and durable SQLite storage. The twelve-stage
pipeline shape, the four-protocol intent preservation chain, and the structured refusal envelope are
architectural artifacts not present in other open-source guardrail libraries surveyed in Chapter 1.
Methodological contributions. The dissertation demonstrates an audit-driven development approach: a
self-conducted architecture audit at the midterm review identified seventeen issues across four severity
levels with concrete file:line references; the second-phase work executed against this audit and reports a
94% closure rate (sixteen of seventeen issues resolved, one partial). This methodology, explicit audit,
traceable remediation, honest disclosure of residual risk, is a transferable contribution that applies to any
production-bound software project.
Empirical contributions. The 137-prompt threat corpus (nine threat categories, three difficulty
gradients, two baselines) is a reusable evaluation artifact for future work in this space. The per-stage
latency decomposition (Table 11) provides a budget breakdown that operators can use to identify
bottlenecks in their own deployments. The codebase health scorecard at midterm (B+) and postremediation (A−) provides a quantitative measure of the remediation impact.
The structural risk of enterprise AI adoption, that business-sensitive context flows to third-party
inference endpoints under contractual rather than technical guarantees, is not addressed by stronger
contracts. It is addressed by detection, transformation, and audit at the boundary between the application
and the model. The engine described in this dissertation is a working implementation of that boundary:
open, extensible, instrumented, and honest about what it does and does not guarantee.

6.3 Recommendations
For organizations considering adoption of the engine, three deployment recommendations emerge from
the work.
48

Use embedded library mode for latency-sensitive paths. The protocol-first design means an
organization can install arc-guard-core plus the runtime extras they actually need, without paying
transitive dependency cost for unused features. Embedded deployment incurs zero per-request network
overhead, the difference between sub-25 ms lite mode and 5–15 ms additional network-hop latency in
centralized mode is operationally significant for high-throughput applications.
Run lite mode in production paths and full mode in audit paths. The engine's lite mode (semantic
and ML-jailbreak inspectors disabled) is appropriate for high-volume production paths where latency
dominates. Full mode is appropriate for audit replay or high-risk paths where additional detection is
worth the latency. The FlagProvider enables this split without code changes.
Invest in the lifecycle sink early. The default SqliteLifecycleSink is sufficient for singleinstance deployments and supports the dashboard out of the box. Multi-instance deployments should
either share a SQLite path on networked storage (operationally simple but introduces a shared-storage
failure mode) or migrate to PostgreSQL via the optional [postgres] extra (more operationally
complex but production-grade). The third option, federated SQLite stores with a cross-instance index, is
forward work.
For organizations evaluating the engine against compliance requirements: the engine supports GDPR
data minimization through redaction, PCI scope reduction through credit card detection, and DPDPA
jurisdictional compliance through the custom entity registry (e.g., Aadhaar). Certified PCI tokenization
is not delivered by the engine itself; the TokenizationProvider Protocol provides the integration
seam for a certified vault, but the certification itself is the operator's responsibility.

6.4 Future Work
Five forward-looking themes emerge for post-submission work, each tied to a specific gap in the current
submission.
49

Multi-tenant deployment. The current default lifecycle sink is single-writer SQLite. Forward work
extends the LifecycleSink Protocol to a multi-instance deployment via either networked SQLite
(e.g., dqlite or rqlite for Raft-replicated SQLite) or a fully-validated PostgreSQL implementation.
Per-tenant isolation, separate audit stores per organization in a multi-tenant deployment, is a related
extension that requires a tenant_id field on LifecycleEvent and tenant-scoped query routes on
the dashboard.
Certified tokenization integration. The TokenizationProvider Protocol scaffolded during
remediation requires concrete integration with at least one certified tokenization vault. Candidate
integrations include AWS Payment Cryptography, HashiCorp Vault Transform Secrets Engine, or a
hardware security module via PKCS#11. The integration must be validated against PCI-DSS audit
requirements, including key-rotation procedures, audit-log immutability, and separation of duties
between key custodians and operational staff.
Threat corpus expansion via red-team exercise. The current 137-prompt corpus is curated; production
deployments encounter attack distributions not represented in curated test sets. Forward work conducts a
structured red-team exercise across at least three target deployments, captures the resulting attack
corpus, and feeds the corpus back into the evaluation harness. The exercise should specifically target the
architecturally-novel detection paths introduced during remediation: multi-turn deception trajectories,
intent-preservation fidelity gates, and structured refusal envelopes.
Multi-language inspector parity. The current implementation is pure Python. The STAGE_CLASSIFY
stage dominates full-mode latency (67% of P99 budget per Table 11), of which the majority is the
DistilBERT semantic inspector. Forward work investigates a Rust or Go implementation of the regex
injection, SQL injection, and shell injection inspectors, the pattern-matching inspectors with the most
predictable performance characteristics, exposed to the Python pipeline via PyO3 or ctypes. The

50

Protocol-first extension model makes such an extension straightforward: a Rust inspector that satisfies
the Inspector Protocol via a Python adapter is structurally equivalent to a Python inspector at the
type level.
Federated lifecycle audit. Multi-region deployments produce per-region lifecycle stores that cannot
trivially be queried as a unified set. Forward work designs a federated audit query layer that supports
cross-region "find request rid=X" queries without centralizing the per-region storage.
Three open research questions also emerge from the dissertation that warrant academic rather than
engineering investigation. First, the cosine-similarity threshold of 0.78 used for fidelity verification was
selected empirically; a principled derivation tied to information-theoretic bounds on response-intent
divergence would strengthen the design. Second, the multi-turn deception detection currently relies on
hand-engineered trajectory patterns; a learned trajectory classifier would generalize more robustly to
novel attack patterns but introduces training-data and bias concerns that the current system avoids by
construction. Third, the relationship between detection rate and false positive rate is not formally
characterized; an ROC analysis across the threat corpus and a benign baseline corpus would quantify the
operating point of the system in a way that the current categorical detection-rate table does not.

cleanly); medium is obfuscated, paraphrased, or indirect (some variants are documented as expected
gaps); super_hard is legitimately framed, red-team wrapped, or multi-layer context (blocked if the
requested output is itself the harmful payload).

54

Threat categories: prompt injection, PII/PCI, jailbreak heuristic, jailbreak ML, multi-turn deception,
semantic policy violation, SQL injection, shell injection, template injection (SSTI). Detailed mapping is
given in Table 6.
Baseline categories: _baseline__multi_turn (validates that benign multi-turn conversations
pass without findings) and _baseline__no_inspector (validates that benign single-turn requests
pass without findings).
The corpus is regenerated as new threat categories are added in subsequent phases of the project; the
current snapshot reflects the post-midterm submission state.

Appendix D: Configuration Reference
This appendix consolidates the runtime configuration surface of the engine. The configuration is layered
in three tiers, each with a distinct lifecycle: structural configuration (Pydantic v2 models loaded from
environment variables at startup), behavioral configuration (Unleash feature flags resolved per-request
through the FlagProvider Protocol), and operational configuration (NATS subject names, OTel
endpoints, sink storage paths). The separation between structural and behavioral configuration is
deliberate: structural settings control what the engine can do (which inspectors are wired in, which
strategies are registered), while behavioral settings control how the engine should behave right now
(whether to short-circuit on the first finding, which risk threshold to apply, which percentage of traffic
gets the experimental policy). Operators can change behavior at runtime by editing flags without
restarting the service.

This appendix documents the reproduction procedure for the engine. It is included so that an operator or
examiner can take the public repository and reproduce a running instance with the threat corpus passing
against it.
57

Repository layout: the engine is a monorepo with three Python packages under packages/. core/

contains the runtime (arc-guard) with Presidio, DistilBERT, NATS, OpenTelemetry, and Unleash as
optional extras. api/ contains the FastAPI service (arc-guard-service) with the threat corpus
and the end-to-end test suite. Each package has its own pyproject.toml with explicit version
